# 旧前端迁移到 React + Vite 框架执行方案 v2

## 1. 项目背景

这份文档定义 `HBweb` 旧前端迁移到新前端框架的执行方案。目标
不是重写业务，而是在保留现有业务能力的前提下，把旧前端逐步迁入
`react-vite-admin-phase1`，并与
`d:\DevRepos\HBweb-worktrees\Backend\publish-master` 的后端接口重新打通。

当前已有三套上下文：

- 旧前端项目：`React + Umi`
- 后端项目：`.NET / ASP.NET Core Web API`
- 新前端框架：`React + Vite + TypeScript + Ant Design + react-router-dom + zustand + keepalive-for-react`

本次迁移的基本原则如下：

- 保留旧前端的大部分业务页面内容
- 去掉 Umi 运行时耦合
- 统一迁入当前新的 `React + Vite` 后台框架
- 统一使用新的左侧导航、Tab 管理、KeepAlive 缓存机制
- 优先复用旧的服务层、类型定义、业务组件和页面逻辑

## 2. v2 方案目标

v2 不再停留在“迁移方向说明”，而是明确执行顺序、前置条件、验收标准
和第一批落地页面。执行过程中必须先固化认证与接口基线，再迁具体业务
页面，避免迁一半后反复返工请求层和权限层。

最终目标如下：

- 技术栈统一为 `React + Vite + TypeScript`
- 布局统一为新框架后台布局
- 菜单统一由新路由元数据生成
- Tabs 统一由路由驱动
- 页面缓存统一由 `KeepAlive` 处理
- 详情页统一支持独立 Tab
- Tab key 统一使用完整路径
- Tab 标题支持动态更新
- 认证、权限、请求层与后端重新打通
- 旧业务页面按模块平滑迁入

## 3. 当前基线

在继续迁移前，以下基线已经确认，后续页面必须遵守，不能各自实现一套
新规则。

### 3.1 路由与 Tab 基线

当前新前端已经采用路由元数据驱动菜单、Tab、KeepAlive 和详情页标题。
后续所有业务模块都必须沿用这套模式，不能在页面内部额外维护一套 Tab
系统。

规则如下：

1. 每个详情页单独开 Tab
2. Tab key 使用完整路径
3. 列表页、表单页、详情页默认开启 `keepAlive`
4. 登录页、403、404 不缓存
5. 菜单来源为路由 `meta`
6. 权限判断来源为 `auth store + route meta`
7. 页面缓存 key 与 Tab key 保持一致

示例：

- `/system/users/10001`
- `/system/users/10002`

以上两个路径会打开两个独立 Tab，不共享详情状态。

### 3.2 认证基线

认证相关逻辑已经不是“待确定”，而是当前迁移的硬前置条件。新前端接入
后端时，必须严格遵守以下契约：

1. 登录接口为 `POST /api/Auth/login`
2. 登录密码字段沿用旧前端规则，先做 `SHA-256` 再提交
3. 前端请求必须携带 `credentials: 'include'`
4. 登录成功后必须调用 `GET /api/Auth/current` 获取用户与权限
5. 前端不直接读取 Cookie 中的 token
6. 未登录或 Cookie 失效后统一回到 `/login`

说明：

- 后端当前是 Cookie 登录模式
- 后端 CORS 使用白名单 + `AllowCredentials`
- 旧前端登录契约并不是明文密码
- 权限数据不以内嵌 token 为准，而以 `/api/Auth/current` 为准

### 3.3 后端接口基线

系统管理相关接口已经在后端具备可复用能力，但前端不能只看文档，必须以
控制器、DTO 和服务实现为准。迁移时以真实接口契约为准，不以旧页面表单
字段想当然平移。

当前已确认的接口组如下：

- 认证：`/api/Auth/login`、`/api/Auth/current`、`/api/Auth/logout`、`/api/Auth/refresh`
- 用户：`/api/Users/**`
- 角色：`/api/Roles/**`
- 分店：`/api/Stores/**`

## 4. 总体迁移策略

本次迁移采用“壳子保留、内容迁移、基线先行、样板带动”的策略。也就是
说，先保留现有新前端壳子，再逐步把旧前端页面、服务层和类型迁入，而不
是把旧工程整体搬过来。

策略拆解如下：

- 保留当前已经搭好的 `React + Vite` 新框架
- 菜单、Tab、KeepAlive、布局全部沿用新框架
- 旧前端页面内容逐步搬入
- 旧前端服务层逐步搬入
- 旧前端类型和通用组件逐步搬入
- 与 Umi 绑定的部分全部剥离
- 每迁完一个页面组就联调、验收、再进入下一组

这套策略的好处如下：

- 风险低
- 可以分阶段上线
- 每迁完一个页面组就能验证
- 不会被 Umi 运行时牵制
- 可以尽早暴露接口字段和权限问题

## 5. 目标目录与模块组织

目录结构仍然以当前新前端为基础，不为迁移单独发明第二套结构。新增业务
模块优先按“页面组 + 服务 + 类型”的方式组织。

建议目录如下：

```txt
src
├── components
├── hooks
├── layout
├── pages
│   ├── Login
│   ├── Dashboard
│   ├── System
│   ├── Warehouse
│   ├── Products
│   ├── StoreOrder
│   └── Reports
├── router
├── services
├── store
├── styles
├── types
└── utils
```

路由负责以下职责：

- 菜单显示
- 权限控制
- Tab 标题
- KeepAlive 开关
- 当前选中菜单
- 动态详情页标题

## 6. 分阶段执行计划

v2 的关键改动是把“认证与接口基线”前移为所有业务迁移的硬前置阶段，然
后再用一个首个业务页面组验证整套迁移方法。

### 阶段 0：认证与接口基线固化

这一阶段的目标是把认证、请求层、环境配置和接口契约收口，确保后续页面
迁移不会反复修改登录态和请求封装。

需要完成：

1. 固化登录密码 `SHA-256` 提交规则
2. 固化 `credentials: 'include'` 请求方式
3. 固化登录后拉取 `/api/Auth/current` 的流程
4. 确认开发和生产环境 API 地址策略
5. 确认 Cookie、CORS、SameSite、Secure 的联调要求
6. 统一请求错误提示和 401 处理策略
7. 冻结系统管理与首批业务页面所需接口字段

阶段验收标准如下：

- 登录稳定
- 刷新页面后能恢复登录态
- 失效后能正确跳回登录页
- 接口错误能正确提示
- 开发环境与后端联调稳定

### 阶段 1：系统管理样板稳定化

这一阶段的目标不是“继续扩大范围”，而是把当前已落地的系统管理页面当作
新框架样板，把列表页、详情页、动态 Tab 标题、权限控制和请求模式先跑顺。

当前已具备的样板内容如下：

- 用户列表与详情
- 角色列表与详情
- 分店列表与详情
- 动态详情页独立 Tab
- 当前用户状态
- 权限过滤

这一阶段要继续确认的点如下：

1. 列表页筛选、分页、跳详情模式
2. 详情页独立 Tab 的统一实现
3. 页面返回后不丢状态
4. 动态标题更新时机
5. 页面错误态和空态的统一表现

阶段验收标准如下：

- 系统管理样板可作为业务页面模板复用
- 所有详情页支持动态 Tab
- 页面刷新、切换不丢状态
- 权限控制和菜单高亮行为稳定

### 阶段 2：首个业务页面组迁移

这一阶段不再写成“泛泛的高频业务模块迁移”，而是明确选择仓库管理导航下
的 3 个页面，作为首个业务页面组迁移对象。目标是用真实业务页面验证新框
架在路由、Tabs、KeepAlive、权限和动态标题上的接入方式。

本阶段优先迁移的页面如下：

1. 仓库商品
2. 分店订货列表
3. 仓库标签管理

这三个页面的选择原因如下：

- 都是旧前端仓库管理一级导航下的直接页面
- 页面职责清晰，适合拆解迁移
- 能覆盖列表、详情、弹窗、辅助数据请求等常见模式
- 能验证业务页面在新框架中的真实接入成本

阶段验收标准如下：

- 仓库管理 3 个页面可在新框架中独立访问
- 菜单、高亮、面包屑、Tabs 行为正确
- 至少一个详情页支持独立开 Tab 和动态标题
- 页面切换后缓存行为符合当前 KeepAlive 规则
- 页面调用真实后端接口并完成联调

### 阶段 3：批量业务模块迁移

在仓库管理 3 个页面跑通后，再进入批量业务模块迁移。此时再扩展到其他业
务模块，成本会更低，因为路由接入、请求层适配、页面模板和详情页模式都
已经有样板。

推荐进入顺序如下：

1. 仓库管理剩余页面和详情页
2. 商品相关模块
3. 分店订货相关剩余流程页
4. 价格与促销相关模块
5. 报表与看板

### 阶段 4：复杂模块迁移

这一阶段处理耦合高、流程长、依赖旧运行时较重的页面。此阶段的重点不是
照搬，而是模块级重构、逻辑拆分和性能优化。

通常包括：

- 仓储复杂流程页
- 发票模块
- 货柜模块
- 大表格或虚拟表格页面
- 多步骤表单
- 复杂统计和看板页面

### 阶段 5：收尾与统一

最后阶段负责清理迁移过程中的兼容代码、临时适配层和重复实现，保证新项
目后续可以独立维护。

需要完成：

1. 统一代码风格
2. 统一目录结构
3. 删除旧兼容代码
4. 删除临时过渡组件
5. 增加环境配置说明
6. 增加迁移文档
7. 增加部署文档
8. 增加联调说明

## 7. 首个业务页面组范围

这一节明确阶段 2 的迁移范围，避免后续把“仓库管理”理解成整个大模块一次
性搬迁。

### 7.1 页面清单

首批页面清单如下：

| 页面名称 | 旧前端路由 | 旧前端页面文件 | 旧前端主服务 |
| --- | --- | --- | --- |
| 仓库商品 | `/warehouse/products` | `src/pages/WareHouse/WarehouseProducts/index.tsx` | `src/services/productWarehouse.ts` |
| 分店订货列表 | `/warehouse/store-orders` | `src/pages/WareHouse/StoreOrder/OrderList/index.tsx` | `src/services/storeOrder.ts` |
| 仓库标签管理 | `/warehouse/locations` | `src/pages/WareHouse/LocationManagement/index.tsx` | `src/services/location.ts` |

补充说明：

- `分店订货详情` 在旧前端是隐藏菜单页面，后续作为第二批仓库页面处理
- 首批不要求一次性覆盖仓库模块全部旧功能
- 首批重点是接入模式验证，不是一次性功能对等

### 7.2 页面组迁移顺序

仓库管理 3 个页面建议按以下顺序迁移：

1. 先迁 `仓库标签管理`
2. 再迁 `仓库商品`
3. 最后迁 `分店订货列表`

顺序原因如下：

- `仓库标签管理` 页面相对收敛，适合先打通业务菜单接入
- `仓库商品` 能验证列表页、弹窗、辅助数据加载
- `分店订货列表` 会带出详情、拣货单、发票等后续链路，适合放在第三步

### 7.3 每个页面的迁移动作

每个页面都按同一节奏处理，避免不同页面采用不同方式改造。

执行步骤如下：

1. 先梳理旧页面的路由、接口、类型和依赖组件
2. 再确定新路由 `meta`、菜单位置和权限 key
3. 再迁服务层和类型定义
4. 再迁页面主体 UI 和筛选条件
5. 再补详情页、弹窗、导入导出等辅助能力
6. 最后联调、修正字段差异并做验收

## 8. 详细执行顺序

下面的顺序用于真正开始编码时的执行清单。只有当前一步完成并验收后，才
进入下一步。

1. 固化认证与请求层基线
2. 复核系统管理样板是否可复用
3. 为 `Warehouse` 分组补路由规划和权限规划
4. 迁移 `仓库标签管理`
5. 迁移 `仓库商品`
6. 迁移 `分店订货列表`
7. 处理 `分店订货详情` 等隐藏页面
8. 总结仓库页面组迁移模板
9. 再进入下一批业务模块

## 9. 页面开发规范

从阶段 2 开始，所有业务页面统一按下面的规范接入新前端。

### 9.1 路由规范

所有业务页面都必须在 `src/router/routes.tsx` 中显式注册，并由路由 `meta`
驱动菜单、权限、缓存和动态标题。

示例：

```ts
{
  path: '/warehouse/locations',
  element: <WarehouseLocationPage />,
  meta: {
    title: '仓库标签管理',
    keepAlive: true,
    accessKey: 'Warehouse.Location.View',
  },
}
```

详情页示例：

```ts
{
  path: '/warehouse/store-orders/:id',
  element: <StoreOrderDetailPage />,
  meta: {
    title: '分店订货详情',
    hidden: true,
    keepAlive: true,
    activeMenu: '/warehouse/store-orders',
    dynamicTitle: (params) => `分店订货详情 - ${params.id || ''}`,
  },
}
```

### 9.2 状态规范

状态管理必须分层，不把所有数据都塞进全局 store。

约定如下：

- 页面临时状态使用 `useState`
- 页面切换保活交给 `KeepAlive`
- 跨页面共享状态交给 `zustand`
- 可恢复筛选条件尽量放到 URL query

### 9.3 Umi 依赖替换规范

旧前端页面迁入时，重点替换以下 Umi 依赖：

- `@umijs/max`
- `history`
- `useModel`
- `useAccess`
- `useRequest`
- runtime layout 配置
- `.umirc.ts` 路由定义
- `getInitialState`

替换方向如下：

- `history` -> `useNavigate`
- `useModel` -> `zustand / hooks / props`
- `useAccess` -> `authStore.access`
- `useRequest` -> `request + useEffect`
- `.umirc.ts` -> `src/router/routes.tsx`

## 10. 联调方案

联调方案直接服务于阶段 0 和阶段 2，重点是把 Cookie 登录与业务接口访问
稳定下来。

### 10.1 开发环境

推荐联调方式如下：

- 后端在本机运行：`http://localhost:5001`
- 前端在本机运行：`http://localhost:5173`
- Vite 代理：`/api -> http://localhost:5001`

这套方式的好处如下：

- 不需要前端处理复杂 CORS
- Cookie 登录联调更方便
- 与当前新前端请求封装方式兼容

### 10.2 生产环境

生产环境推荐前后端统一域名或统一网关，减少跨域 Cookie 问题。前端如果变
更域名，必须重新确认后端 CORS 白名单和 Cookie 策略。

## 11. 风险与处理策略

迁移过程中最容易出问题的不是页面本身，而是运行时依赖、接口字段和详情
页交互方式。下面列出 v2 重点关注的风险。

### 风险 1：旧页面强依赖 Umi runtime

处理方式如下：

- 按页面组替换
- 不一次性全搬
- 优先迁服务层和类型，再迁页面

### 风险 2：认证契约被误改

处理方式如下：

- 固化登录密码 `SHA-256` 规则
- 固化 Cookie 登录方式
- 固化 `/api/Auth/current` 为权限来源

### 风险 3：接口字段和旧页面表单不一致

处理方式如下：

- 先冻结首批页面接口字段
- 以控制器和 DTO 实现为准
- 请求层统一做适配，不在页面内零散转换

### 风险 4：详情页打开逻辑杂乱

处理方式如下：

- 强制统一成完整路径开 Tab
- 禁止页面内部自定义第二套路由或 Tab 机制
- 隐藏菜单详情页统一用 `activeMenu` 回指列表页

### 风险 5：列表页筛选状态丢失

处理方式如下：

- `KeepAlive + URL query` 结合
- 列表页默认保活
- 详情返回依赖浏览器路由和缓存恢复

### 风险 6：旧页面体积过大

处理方式如下：

- 模块级懒加载
- 动态 `import`
- 合理拆包
- 先拆组件再迁页面

## 12. 验收标准

每个迁移页面组都按同一标准验收，避免“能打开就算完成”。

验收清单如下：

1. 页面能打开
2. 菜单能正确定位
3. Tab 能正确打开
4. Tab 能正确关闭
5. 详情页能独立开 Tab
6. 标题能动态更新
7. 返回后不丢状态
8. 权限控制正确
9. 刷新页面后仍能恢复
10. 接口报错时有正确提示
11. 与真实后端接口联调通过

## 13. 当前建议

结合当前进度，下一步不再继续泛化讨论“高频模块优先级”，而是直接以仓库
管理 3 个页面作为首个业务页面组开始执行。

推荐的实际推进顺序如下：

1. 继续稳定认证与请求层
2. 复核系统管理样板
3. 先迁 `仓库标签管理`
4. 再迁 `仓库商品`
5. 最后迁 `分店订货列表`
6. 完成后沉淀 `Warehouse` 页面组模板

## 14. 核心原则

最后把本次迁移收口成 7 条原则，后续新增页面默认遵守：

1. 不重做业务，优先迁移
2. 不保留 Umi 运行时依赖
3. 认证和接口基线先于业务页面迁移
4. 路由统一驱动菜单和 Tabs

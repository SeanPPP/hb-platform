## 问题分析

* 当前方法使用 SqlSugar 的 LINQ 风格 `OrderBy/OrderByDescending`，在与 `Skip/Take` 分页及 `Select` 中包含子查询 (`SqlFunc.Subqueryable`) 同时存在时，部分版本/驱动下会出现 ORDER BY 被忽略或不稳定的情况，导致返回数据未按日期排序。

* 代码先 `CountAsync()` 再构造排序查询；虽然一般无影响，但推荐用 `Clone()` 计算总数，避免查询状态被复用带来的潜在副作用。

* 现有排序只按主键字段单一排序，缺少二级排序键，遇到同一天的多条记录时结果不稳定。

## 修复方案

1. 统一使用 SqlSugar 的 `OrderBy(expr, OrderByType)` API 替代 LINQ 风格，确保在分页 SQL 中稳定生成 `ORDER BY`。
2. 为日期排序增加二级排序键、`OrderNo`，保证同日多记录的确定性顺序。
3. 对 `SortBy` 做 `Trim().ToLower()` 的归一化，兼容大小写与可能的前后空格；默认仍为 `orderdate`。
4. 将 `total = await q.CountAsync()` 改为 `total = await q.Clone().CountAsync()`，避免对后续查询链的影响。
5. 其他排序分支（金额、数量、状态等）全部改用 `OrderBy(expr, OrderByType)`；其中“总数量”分支继续用子查询但采用 `OrderBy(..., orderType)`。
6. 可选：增加接口级调试日志开关，输出生成的 SQL 便于验证（不默认开启）。

## 代码改动要点

* 定义 `var orderType = (filter.SortDescending ?? true) ? OrderByType.Desc : OrderByType.Asc;`

* `orderdate` 分支：`orderedQuery = q.OrderBy(o => o.OrderDate, orderType).OrderBy(o => o.OrderNo, orderType);`

* 其他分支示例：`orderedQuery = q.OrderBy(o => o.ImportTotalAmount ?? 0, orderType);`

* “总数量”分支：`orderedQuery = q.OrderBy(o => SqlFunc.Subqueryable<WareHouseOrderDetails>().Where(d => d.OrderGUID == o.OrderGUID).Sum(d => d.Quantity) ?? 0, orderType);`

* 计算总数：`var total = await q.Clone().CountAsync();`

## 原因分析
- 套装商品场景：条码列已使用 `Input`，但键值使用不一致（`setProductCode`/临时 `tempId`）或受控值为 `undefined` 时可能导致输入无效感知；同时表格重绘会打断中文输入法合成（IME），造成“焦点有但无法输入”。
- 多码商品场景：当前条码列为 `Button link`，不可编辑，容易被误认为无法输入。

## 修复方案
- 统一行键与编辑键：在渲染函数中先计算 `rowKey = row.setProductCode ?? row.tempId`，所有受控值与 `onChange` 都使用该键，避免键不一致导致受控值不更新。
- 保证受控值为字符串：对 `value` 做 `?? ''` 兜底，杜绝 `undefined` 受控值，在首次输入时确保变更生效。
- 兼容中文输入法：为条码 `Input` 增加 `onCompositionStart/onCompositionEnd`，合成期间把值写入本地（不触发保存），合成结束再更新编辑 map，避免 IME 被重绘打断。
- 交互提示：
  - 套装商品：保留“添加条码”按钮；新增行默认聚焦到条码 `Input`（设置 `autoFocus`），提升可编辑性。
  - 多码商品：条码列改为只读展示但加说明“多码条码需后端支持编辑”，避免误解；或者临时提供 `Input` 但禁用保存，仅用于前端预览（推荐保留只读）。

## 前端具体改动
1. 在套装子表列渲染中使用统一键并兜底：
```
const rowKey = row.setProductCode || row.tempId;
<Input
  value={(setSetItemsEdits[rowKey]?.setBarcode ?? row.setBarcode ?? '')}
  onChange={(e) => setSetItemsEdits(prev => ({
    ...prev,
    [rowKey]: { ...prev[rowKey], setBarcode: e.target.value }
  }))}
  onCompositionStart={() => setIsComposing(true)}
  onCompositionEnd={(e) => { setIsComposing(false); /* 同步最终值 */ }}
  autoFocus={Boolean(row.tempId)}
/>
```
2. 价格列同样使用 `rowKey` 并对 `value` 做 `?? undefined` 兜底，确保受控。
3. 多码商品条码列维持只读 `Button link`；在列 `title` 或空态提示上增加说明，避免误操作。

## 保存逻辑
- 套装商品：遍历 `setItems` 生成提交项，使用统一键合并编辑值；新行（无 `setProductCode`）也包含在提交中，由后端创建。
- 多码商品：维持仅价格提交；条码不提交（后端不支持）。

## 验证
- 新增行后条码 `Input` 自动聚焦，可输入中文与数字，输入过程中不丢失；保存后刷新弹窗数据验证新条码生效。
- 既有行修改条码与价格均可保存；多码商品仅价格可改，条码只读不再误导。

## 影响范围
- 仅编辑弹窗内部的子表，不影响主列表与其他页面。

## 需求概述
- 定义“上浮率表”，按进货价区间映射到上浮率：例如区间 [1,10] 对应上浮率从 2.5 逐步过渡到 2.2。
- 支持两种插值模式：等分线性（线性插值）与指数级（曲线更平滑且可控）。
- 自动零售价 = 进货价 × 上浮率；在开启自动定价的场景中（`IsAutoPricing=true`），系统根据上浮率表自动生成零售价。

## 插值模型设计
- 变量约定：
  - `L`=下限，`U`=上限，`x`=进货价，`R_low`=下限对应上浮率，`R_high`=上限对应上浮率，`t=(x-L)/(U-L)`（规范化到 [0,1]）。
- 线性插值（等分）：
  - `Rate(x) = R_low - t * (R_low - R_high)`（当 `R_low>R_high` 时随价上升而递减）。
  - 边界：`x=L → R_low`，`x=U → R_high`。
- 指数插值（可控曲率，端点严格匹配）：
  - 令曲率参数 `k>0`，定义归一化曲线 `g(t) = (exp(-k*t) - exp(-k)) / (1 - exp(-k))`，满足 `g(0)=1`、`g(1)=0`。
  - `Rate(x) = R_high + (R_low - R_high) * g(t)`。
  - `k` 越大，曲线越“前倾”（高价段更接近 `R_high`）。若需“后倾”，可允许 `k<0`，同理可得对称效果。
- 多区间/分段：
  - 支持配置多条规则，按区间拼接形成分段函数；每段独立选择插值类型与曲率。
  - 若价格落在多个重叠区间，按 `Priority` 选择优先规则。

## 数据结构与存储（SqlSugar）
- 新表：`StoreMarkupRule`
  - 字段：`Id`、`StoreId`、`LowerBound`、`UpperBound`、`MinRate`（=R_low）、`MaxRate`（=R_high）、`InterpolationType`（`Linear`/`Exponential`）、`CurveK`、`Priority`、`Enabled`、可选：`CategoryId`、`SupplierId`。
  - 约束：同一 `StoreId` 下建议避免区间重叠或通过 `Priority` 明确取舍；`LowerBound < UpperBound`，`MinRate>0`，`MaxRate>0`。
  - 索引：`(StoreId, LowerBound, UpperBound, Enabled)`；可选 `CategoryId`、`SupplierId` 辅助索引。
- 在 `BlazorApp.Api/Data/SqlSugarContext.cs:InitTables()` 与 `PostgresSqlSugarContext.cs:InitTables()` 注册新表。

## 后端服务设计
- 新服务：`MarkupRateService`
  - 核心方法：`GetRate(storeId, purchasePrice, categoryId?, supplierId?)` → 查找匹配规则并按插值类型计算上浮率；无匹配时返回默认（例如 1.0 或站点级缺省）。
  - 规则选择：
    - 过滤：`Enabled=true`、价格落在 `[LowerBound, UpperBound]`；按 `SupplierId/CategoryId` 精确匹配优先；若多条匹配按 `Priority`、再按区间宽度、最后按创建时间。
  - 缓存：
    - 规则按 `StoreId` 缓存（本地内存 + 失效通知），保证发票与列表页面查询性能。
- 暴露 React API：`BlazorApp.Api/Services/React/MarkupRuleReactService.cs`（CRUD、分页、筛选、导入导出）。

## 示例与校验
- 示例：`L=1,U=10,R_low=2.5,R_high=2.2`
  - 线性：
    - `x=1 → 2.5`，`x=10 → 2.2`，`x=5 → 2.5 - (4/9)*0.3 ≈ 2.3667`
  - 指数（k=2）：
    - `t=4/9≈0.444...`，`g(t)=(e^{-0.8889}-e^{-2})/(1-e^{-2})≈0.566`，`Rate(5)≈2.2 + 0.3×0.566 ≈ 2.3698`
- 自动生成零售价：`Retail = Purchase × Rate`；例如 `Purchase=5 → Retail≈11.8335（线性）`。

## 风险与守护
- 配置冲突：区间重叠与优先级冲突 → 启用保存前检测与可视化提示。
- 极端值：负价或零价输入 → 拦截；超出所有区间 → 采用默认策略或最近区间外推（建议默认）。
- 性能：大店规则较多时需缓存与分页；避免在发票批量生成中重复计算可使用批量查询与向量化插值。

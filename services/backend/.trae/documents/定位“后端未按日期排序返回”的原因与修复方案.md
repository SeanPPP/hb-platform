## 现象复核与可能原因
- 你在数据库直接执行 ORDER BY OrderDate DESC, OrderNo DESC 的 SQL，结果正确；但接口返回的 items 顺序与日期不一致，说明“接口实际执行的 SQL”与“我们期望的排序”存在差异，或前端对返回结果做了二次处理。
- 可能的后端原因：
  - ORM 生成的最终 SQL 与我们在代码里设置的排序不一致（多次 OrderBy 合并、字符串排序未携带别名导致被覆盖等）。
  - 实际请求的 filter.SortBy 未按预期传到服务（但从控制器代码看是原样传递）。
- 可能的前端原因：
  - 前端对 items 做了二次排序或插入合并（例如按门店或其他键重排）。
  - 前端以字符串比较日期（而不是 Date 值），导致顺序异常。

## 定位方案
1. 在 GetOrderListAsync 里为这条查询启用 SqlSugar AOP 日志，打印“实际执行的 SQL 与参数”。重点确认 ORDER BY 是否为 "OrderDate DESC, OrderNo DESC"。  
2. 将字符串排序改为显式带别名的组合排序："o.OrderDate DESC, o.OrderNo DESC"，避免潜在歧义。  
3. 暂时增加一个调试开关（例如 filter.Remarks == "debug"），当开启时额外返回“服务器端已执行的 SQL 字符串”，便于你在页面日志对比。
4. 在控制器中打点当前接收到的 filter（sortBy、sortDescending、起止日期），确认入参无误。
5. 前端联调建议：在发起请求处关闭任何客户端排序，将服务器返回的 items 原样展示，验证问题是否消失。

## 问题复核
- 用户仍然看到按 OrderDate 降序不稳定，出现 12-04 被排在 12-02/12-01 之后的情况。
- 可能是部分 SqlSugar/LINQ 组合在分页与投影并含子查询时，生成的多次 OrderBy 合并不稳定。

## 改进方案
1. 在 `orderdate` 分支改用一次性字符串组合排序：`OrderDate DESC, OrderNo DESC`，避免多次 `OrderBy` 合并导致的 ORDER BY 丢失或覆盖。
2. 默认分支同样使用字符串组合排序，保证未指定 SortBy 时也稳定按日期+单号降序。
3. 保留其他分支的 typed OrderBy，但确保不影响分页与 Select 的顺序。
4. 保持总数计算用 `q.Clone().CountAsync()`。

## 验证
- 用 { sortBy: "OrderDate", sortDescending: true } 请求第一页，检查日期是否从最新到最旧，同一天按单号降序。
- 快速编译验证，确保无新增错误，并确认排序逻辑在代码层面强制为单次 ORDER BY。

## 变更范围
- 仅修改 `StoreOrderReactService.GetOrderListAsync` 方法中的 `orderdate` 与默认分支排序实现；不变更接口签名或返回字段。

## 错误定位
- 错误来源：`ApiResponse<BatchResultDto>.Error("批量失败", "BATCH_UPSERT_ERROR")` 出自服务方法的事务捕获块。
- 控制器位置：`d:/Development/cline/blazor/BlazorApp.Api/Controllers/React/ReactLocalSupplierInvoicesController.cs:119-137`（BatchUpsertDetails 动作）。
- 服务方法：`d:/Development/cline/blazor/BlazorApp.Api/Services/React/LocalSupplierInvoicesReactService.cs:428-590`。

## 根因分析
- 当前更新路径使用 `db.Updateable<StoreLocalSupplierInvoiceDetails>(u)` 并在其上链式 `SetColumnsIF(...).Where(x => x.DetailGUID == u.DetailGUID)`. 在 SqlSugar 中，`Updateable(entity)` 通常依赖实体主键；叠加 `Where` 使用可能在某些配置下触发“主键缺失/不匹配”或导致内部生成的 SQL 与事务环境不兼容，从而抛异常进入事务 `catch` 并返回 `BATCH_UPSERT_ERROR`。
- 另一个风险是收到的 `DetailGUID` 无记录时，更新影响行数为 0；目前无回落处理，若内部某一步抛异常同样返回错误。

## 修复方案（服务层）
1. 将更新构造改为“无实体 Updateable”：
   - 使用 `db.Updateable<StoreLocalSupplierInvoiceDetails>()`，逐项从 `u` 显式映射到列（`SetColumnsIF`）并 `Where(x => x.DetailGUID == u.DetailGUID)`。
   - 避免 `Updateable(entity)` 对主键的隐式要求。
2. 增加“更新回落插入”：
   - 执行更新返回 0 行时，直接将该条转为插入，保障批量不会因单条不存在而失败。
3. 强化异常消息：
   - 在 `catch (Exception exTran)` 中将 `exTran.InnerException?.Message ?? exTran.Message` 写入返回的 `message`，便于前端展示与定位。
4. 输入校验：
   - `items == null || items.Count == 0` 时直接返回 `OK({Inserted=0,Updated=0,Failed=0})` 而非事务。

## 前端增强
- 保存失败时显示后端返回的 `message`（而非 Axios 的泛化信息），以便定位具体字段或约束问题。
- 粘贴新增行不再携带 `detailGUID`，避免错误走更新路径。

## 验证
- 粘贴新行 + 已存在行混合保存，应返回 `Inserted/Updated` 正确统计，不再出现 `BATCH_UPSERT_ERROR`。
- `DetailGUID` 不存在的行会自动转插入，不影响事务。

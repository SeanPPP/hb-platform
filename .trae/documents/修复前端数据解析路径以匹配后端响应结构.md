# 解决页面无数据及新建失败问题

**问题分析**

1.  **页面无数据 (后端有返回)**
    *   **现象**: 后端有返回数据，但前端表格为空。
    *   **原因**: 前端数据解析路径与后端返回结构不匹配。
        *   **后端返回**: `GridResponseDto<T>`，结构为 `{ success: true, items: [...], total: 10, ... }`。
        *   **前端代码**: `const d = (res as any)?.data?.data`。
        *   **实际情况**: Umi request 拦截器通常已经解包了第一层 `data`。
            *   如果后端 Controller 返回 `Ok(response)`，HTTP 响应体就是 `GridResponseDto` JSON。
            *   如果 Umi 拦截器配置为 `responseInterceptors: [ (response) => response ]`，那么 `res` 包含 `data` 属性。
            *   但是，前端代码尝试去取 `res.data.data`。这暗示前端期望 `ApiResponse<GridResponseDto>` 结构，但 Controller 直接返回了 `GridResponseDto`。
            *   查看 `ReactPricingStrategiesController.cs`：
                ```csharp
                public async Task<ActionResult<GridResponseDto<...>>> Grid(...) {
                    var res = await _service.GetGridAsync(request);
                    return Ok(res); // 直接返回 GridResponseDto
                }
                ```
            *   查看前端 `index.tsx`：
                ```typescript
                const res = await getGrid(req)
                const d = (res as any)?.data?.data // ❌ 错误：多取了一层 data
                ```
                实际上应该取 `res.data` (如果是 axios response) 或直接是 `res` (如果拦截器已解包)。
                考虑到 Umi Max 的 `request` 默认行为，`res` 通常是直接的数据对象（如果配置了 errorConfig 里的 adapter），或者是包含 `data` 的 axios 对象。
                根据 `loadData` 里的 `const d = (res as any)?.data?.data`，它假设返回的是 `ApiResponse` 包装的结构（即 `ApiResponse.Data` 才是 `GridResponseDto`）。但 Controller **直接**返回了 `GridResponseDto`。
    *   **修复**: 修改前端数据获取逻辑，适配直接返回的 `GridResponseDto`。

2.  **新建保存成功但提示失败**
    *   **现象**: 数据已保存入库，前端提示错误。
    *   **原因**: 前端判断逻辑错误。
        *   `createStrategy` 返回 `ApiResponse<PricingStrategyDetailDto>`。
        *   Controller 返回 `Ok(res)`。
        *   前端代码：
            ```typescript
            const res = await createStrategy(...)
            ok = !!res.data?.success // ❌ 问题点
            ```
        *   如果 Umi request 拦截器没有过度包装，`res` 可能就是响应体数据（包含 `success`）。
        *   如果 `res` 是 AxiosResponse，`res.data` 就是 `ApiResponse` 对象。
        *   需要确认 `ApiResponse` 的属性大小写。后端 C# `Success` -> JSON `success` (camelCase)。
        *   **关键点**: `GridResponseDto` 和 `ApiResponse` 结构不同，前者用于列表，后者用于详情/操作。
        *   `PricingStrategyReactService.CreateAsync` 返回 `ApiResponse`。
        *   检查 `ReactPricingStrategiesController.Create`，返回 `ActionResult<ApiResponse<...>>`。
        *   所以对于 Create/Update，后端返回确实是 `ApiResponse`。前端取 `res.data?.success` 理论上是对的（如果 res 是 AxiosResponse）。
        *   但如果 Umi 拦截器已经把 `res` 变成了 response body，那么应该直接取 `res.success`。

**数据结构对比**
*   **Grid 接口**: 返回 `GridResponseDto` `{ success, items, total }`。
*   **CRUD 接口**: 返回 `ApiResponse` `{ success, data, message }`。

**修改方案**

1.  **修复 `loadData` (列表加载)**
    *   由于 Controller 直接返回 `GridResponseDto`，前端不需要取两层 `data`。
    *   修改 `const d = (res as any)?.data?.data` 为 `const d = (res as any)?.data || (res as any)`。
    *   更稳健的写法：检查 `d.items` 是否存在。

2.  **修复 `saveEditor` (保存逻辑)**
    *   确认 `res` 的结构。如果 `res.data` 存在且包含 `success`，则使用 `res.data.success`。否则尝试 `res.success`。
    *   修改判断逻辑：`ok = !!(res.data?.success ?? (res as any).success)`。

3.  **修复 `handleDelete` (删除逻辑)**
    *   同样适配响应结构。

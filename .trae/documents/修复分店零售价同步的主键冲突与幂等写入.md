## 问题分析
- 报错来源：`DataSyncReactService.ProcessSingleStoreAsync` 在批量插入分店零售价时触发 SQL Server 主键冲突（`PK_StoreRetailPrice_UUID`）。代码位置：`BlazorApp.Api/Services/React/DataSyncReactService.cs:511-527`，调用封装的批量插入：`RetryBulkInsertAsync`（`DataSyncReactService.cs:569-595`）。
- 当前映射：每条价格记录的 `UUID` 由映射配置随机生成（`ReactStoreRetailPriceMappingProfile.cs:12-16`），且表主键仅为 `UUID`（`BlazorApp.Shared/Models/HBweb/StoreRetailPrice.cs:16-17`）。
- 触发原因：
  - 使用随机 `UUID` 作为唯一主键，缺乏“业务幂等”保障；同一条业务记录每次同步会生成不同主键，只能依赖“先删后插”。在高并发与大批量下，任何历史残留或批内重复都会使整页 `BulkCopy` 失败。
  - 删除策略为“按分店批量删除”（`DataSyncReactService.cs:284-295`），一旦分店集合与历史不完全覆盖、或同页存在重复业务键，随机 `UUID` 将放大冲突概率。
- 旁证：项目内已有业务层批量保存采用业务键查重与 Upsert（`StoreRetailPriceReactService.BatchUpsertAsync`，`StoreRetailPriceReactService.cs:627-744`），说明价格数据的自然唯一键应为 `StoreCode + ProductCode + SupplierCode` 或 `StoreCode + StoreProductCode`。

## 改动方案
- 将写入改为“业务幂等 + 批量 Upsert”，并对批内数据做去重：
  1) 稳定主键：`UUID` 改为基于业务键的确定性生成（建议：`StoreCode + StoreProductCode`；若 `StoreProductCode` 可能为空，回退到 `StoreCode + ProductCode + SupplierCode`）。新增 `UuidHelper.GenerateDeterministic(string key)`，使用 SHA-256/MD5 取 32 位十六进制，保证同一业务键生成的 `UUID` 恒定不变。
  2) 写入策略：替换 `RetryBulkInsertAsync` 为 SqlSugar `Storageable` 分流，`WhereColumns` 使用上述业务键，得到 `InsertList` 与 `UpdateList`，分别走 `Fastest.BulkCopy` 与 `Fastest.BulkUpdate`，更新价格相关字段与状态，保留既有 `UUID`。
  3) 批内去重：在映射后对 `localBatch` 按业务键去重，避免同页重复导致失败。
  4) 删除策略优化：采用 Upsert 后可取消“全量按分店删”；如需清理，改为差集删除（仅删除本分店中不在 HQ 的业务键）。

## 实现步骤
- 新增帮助方法：在 `BlazorApp.Shared/Helper/UuidHelper.cs` 添加 `GenerateDeterministic(string key)`，返回 32 字符十六进制字符串。
- 修改映射：`ReactStoreRetailPriceMappingProfile.cs` 将 `UUID` 映射改为基于业务键的确定性生成，优先使用 `H分店代码 + H分店商品编码`，为空时使用 `H分店代码 + H商品编码 + H供应商编码`。
- 批内去重：在 `ProcessSingleStoreAsync` 映射得到 `localBatch` 后，按业务键分组取最后一条，形成去重列表。
- Upsert 写入：用 `localDb.Storageable(localBatch).WhereColumns(x => new { x.StoreCode, x.StoreProductCode })`（或三键）生成 `InsertList`/`UpdateList`；`Fastest.BulkCopyAsync(InsertList)` 与 `Fastest.BulkUpdateAsync(UpdateList)` 执行；更新列包含 `PurchasePrice`、`StoreRetailPriceValue`、`DiscountRate`、`IsActive`、`IsAutoPricing`、`UpdatedAt`、`UpdatedBy`。
- 日志与统计：保留原有分店级统计与耗时日志，并增加当页去重计数与 Upsert 计数输出。

## 验证与回归
- 选取出现过错误的分店（如 `1014`）单独运行同步，期望日志显示“处理 == 成功，失败 == 0”，且无主键冲突。
- 抽样重复运行同一分店同步两次，验证第二次为纯 `Update`，`AddedCount` 不再增长，`UUID` 保持不变。
- 对随机两个分店并发同步，检查总的 “处理/成功/失败/耗时” 汇总无异常。

## 备选方案
- 若短期不改映射与写入：在 `BulkCopy` 前仅做批内去重，并将“按分店删”改为“全表清空再插入”，作为临时止血，但不具备幂等性与可维护性。
- 中期结构优化：将表主键改为业务组合键（`StoreCode + StoreProductCode`），将 `UUID` 改为普通列，彻底消除随机主键冲突风险。

# 标准 RBAC 动态权限实施方案 (UUID v7 版)

收到，我将严格遵循 **标准 RBAC 模型** 并使用 **UUID v7** 作为主键来执行改造。

## 🛠️ 核心执行计划

### 1. 实体定义 (BlazorApp.Shared)
创建两个新实体，继承自 `BaseEntity` (如果适用) 或使用 UUID v7 主键：

*   **`SysPermission`** (权限定义)
    *   `Id`: **UUID v7** (主键)
    *   `Code`: string (业务唯一键，如 `User.Create`) —— *鉴权时的核心标识*
    *   `Name`: string (显示名，如 "创建用户")
    *   `Category`: string (分类)
*   **`SysRolePermission`** (角色-权限关联)
    *   `Id`: **UUID v7** (主键)
    *   `RoleGuid`: string (关联 `Role` 表)
    *   `PermissionCode`: string (关联 `SysPermission.Code`) —— *使用 Code 关联更易读且迁移稳定*

### 2. 数据库上下文 (Data Access)
*   在 `SqlSugarContext` 中注册 `SysPermission` 和 `SysRolePermission`。
*   **自动迁移**: 确保应用启动时自动创建这两张表。

### 3. 种子数据 (Seeding)
*   创建一个 `PermissionSeeder` 服务。
*   扫描代码中定义的所有静态权限（我们将创建一个 `Permissions` 常量类）。
*   启动时自动将这些权限写入 `SysPermission` 表（如果不存在）。
*   **初始化**: 默认将所有权限赋予 "Admin" 角色。

### 4. 服务层实现 (RoleService)
*   实现 `GetPermissionsAsync`: 返回所有权限供前端选择。
*   实现 `AssignPermissionsToRoleAsync`: 更新 `SysRolePermission` 表。
*   实现 `UserHasPermissionAsync`:
    *   查询: `User` -> `UserRole` -> `Role` -> `SysRolePermission` -> `PermissionCode`。
    *   判断用户是否拥有目标 `Code`。

### 5. 鉴权架构 (Authorization)
*   **策略提供器 (`PermissionPolicyProvider`)**: 自动处理 `[Authorize(Policy = "User.Create")]`。
*   **授权处理器 (`PermissionAuthorizationHandler`)**: 调用 `RoleService` 检查权限。

### 6. 验证
*   我们将选择一个 Controller（如 `UsersController`）将 `Roles="Admin"` 替换为 `Policy="User.Create"` 来验证整套链路。

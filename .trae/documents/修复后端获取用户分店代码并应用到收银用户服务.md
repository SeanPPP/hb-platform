## 问题与结论
- 当前 JWT 未包含 StoreCode 声明，而 CashRegisterUserReactService 通过 Claims 读取 "StoreCode"，导致非管理员用户下 `GetCurrentUserStoreCode()` 返回空。
- 用户-分店关系已存在于数据库表 UserStore（含 IsPrimary）与 Store（含 StoreCode）。应在后端通过数据库查询得到用户的分店代码。

## 改造目标
- 改造 `CashRegisterUserReactService.GetCurrentUserStoreCode()`：优先从数据库根据当前登录用户查询主分店的 StoreCode；失败时再回退到 Claims；管理员角色不限制。
- 使列表/详情/创建/更新等方法中的 StoreCode 过滤在非管理员用户下始终有效。

## 实施步骤
### 1. 获取当前用户GUID
- 新增辅助方法 `GetCurrentUserGuid()`：从 `HttpContext.User` 中读取 `userGuid` 或 `userId`（两者之一已在 JWT 内）。

### 2. 数据库查询用户主分店代码
- 在 `GetCurrentUserStoreCode()` 内：
  - 使用 `GetCurrentUserGuid()` 获取 `userGuid`；
  - 通过 `SqlSugar` 查询 `UserStore` 表：优先 `IsPrimary == true` 的记录；若无主分店，则取该用户第一条分店关联；
  - 通过关联的 `StoreGUID` 查询 `Store` 并读取 `StoreCode`；
  - 返回 `StoreCode`；若查询不到，回退尝试从 Claims 读取 `"StoreCode"`；最终可能返回 `null`。

### 3. 管理员判定与行为
- 使用 `HttpContext.User.IsInRole("Admin")` 或读取角色 Claims；
- 在现有各查询/修改方法中，保持逻辑：`if (!isAdmin && !string.IsNullOrEmpty(userStoreCode)) baseQuery = baseQuery.Where(u => u.StoreCode == userStoreCode);`
- 若非管理员且无分店绑定：统一返回空列表或失败消息（例如 403/业务错误提示），以避免越权。

### 4. 兼容性与回退
- 保留对 Claims 中 `StoreCode` 的回退读取，以兼容后续若增加该声明的情况；但不依赖它。

### 5. 验证点
- 使用一个非管理员且已绑定分店的账号：
  - 列表只返回该分店数据；
  - 创建/更新时仅允许针对该分店；
- 使用未绑定分店的非管理员账号：
  - 列表返回空或提示无分店绑定；
- 使用管理员账号：
  - 返回所有分店数据；

## 变更范围
- 仅修改 `BlazorApp.Api/Services/React/CashRegisterUserReactService.cs`，新增两个私有辅助方法并改造 `GetCurrentUserStoreCode()`；其余方法沿用现有过滤逻辑。

## 备选方案（可选，不在本次实施）
- 在 `AuthService` 的 `GenerateJwtToken`/`GenerateAccessToken` 中加入主分店的 `StoreCode` Claim；可作为性能优化减少一次数据库查询，但需要处理多分店用户的语义。

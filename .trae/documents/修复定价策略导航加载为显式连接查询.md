## 目标
- 使用 SqlSugar 的 `LeftJoin` 将策略、规则、目标扁平化查询，再在内存按 `StrategyId` 分组组装为 `Details/Targets` 集合，彻底绕开导航 Includes 问题。

## 查询设计
- 仅取启用策略：`s.IsEnabled == true`
- 进货价区间预过滤（可选，减少行数）：`purchasePrice >= d.MinPrice && purchasePrice <= d.MaxPrice`
- 扁平化查询：
  - `Queryable<PricingStrategy>`
  - `.LeftJoin<PricingStrategyDetail>((s, d) => s.Id == d.StrategyId)`
  - `.LeftJoin<PricingStrategyTarget>((s, d, t) => s.Id == t.StrategyId)`
  - `.Where(s => s.IsEnabled)`（以及区间 Where 条件）
  - `.Select((s, d, t) => new { s.Id, s.Name, s.Level, s.TargetCode, s.Priority, /* 规则字段 */ d.Id, d.MinPrice, d.MaxPrice, d.StartRate, d.EndRate, d.Algorithm, /* 目标字段 */ t.Id, t.TargetType, t.TargetCode })`

## 组装流程
1. 执行扁平化查询得到 `rows`
2. `GroupBy` 按 `StrategyId`：`var groups = rows.GroupBy(r => r.Id)`
3. 对每个组：
   - 还原策略主字段：`Name/Level/TargetCode/Priority`
   - 组装 `Details = group.Select(r => new PricingStrategyDetail { ... }).Where(distinct by detail.Id).ToList()`
   - 组装 `Targets = group.Select(r => new PricingStrategyTarget { ... }).Where(distinct by target.Id).ToList()`
4. 得到完整的策略集合（每条含 `Details/Targets`）供后续匹配逻辑使用

## 替换位置
- 文件：`BlazorApp.Api/Services/Pricing/AutoPricingService.cs`
  - 新增 `LoadStrategiesWithRelationsByJoinAsync(decimal purchasePrice)`，返回包含 `Details/Targets` 的策略集合
  - 在 `FindStrategyForPriceAsync` 中调用该方法，移除原 `Includes`，并保留等级（Supplier > Store > Global）与 `Priority` 的选择逻辑
- 注意：若不需“进货价预过滤”，可在 `Load...` 中去掉区间 Where，以便返回所有启用策略，然后再在内存过滤区间

## 排序与选择
- 先过滤命中区间的策略（如果未在 SQL 中预过滤）
- 按等级优先：Supplier → Store → Global；同等级按 `Priority` 倒序取首个

## 验证
- 构建后端，调用 `POST /api/react/v1/pricing-strategies/evaluate`，检查响应 `rule` 与 `strategyId` 有值
- 使用前端测试区域输入不同进货价，验证命中策略/规则正确变化

## 备注
- `LeftJoin` 结果会出现笛卡尔乘积，组装时需对 `Details/Targets` 做 `distinct` 去重（按各自 Id）
- 若数据库数据量较大，建议在 SQL 层增加 `purchasePrice` 区间预过滤以减少行数，提高性能

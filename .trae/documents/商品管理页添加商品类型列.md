## 原因分析
- 现有产品 React 查询 DTO 不包含商品类型过滤；服务端排序支持若干字段但缺少 `productType`。
- 前端条码列为纯文本，未提供 Code128 可视化。

## 后端改动（商品类型筛选/排序）
- DTO：在 `BlazorApp.Shared/DTOs/ProductReactDtos.cs` 的 `ProductReactFilterDto` 增加 `public int? ProductType { get; set; }`
- 服务：在 `BlazorApp.Api/Services/React/ProductReactService.cs`
  - 过滤：在构建查询时加入 `if (query.ProductType.HasValue) q = q.Where(p => p.ProductType == query.ProductType.Value);`
  - 排序：在 `switch (sortField)` 增加 `case "producttype":` 分支，按升/降序排序。
- 控制器：`ReactProductController` 无需改动（接收 DTO 自动映射）。

## 前端改动（商品类型列支持筛选/排序）
- 类型：在 `ReactUmi/my-app/src/services/product.ts` 的 `ProductFilterParams` 增加 `productType?: number;`
- 请求：构建 `params` 时若选择了商品类型，传递 `productType`；排序点击“商品类型”列头时传递 `sortBy=productType` 与 `sortOrder`。
- 列定义：在 `ReactUmi/my-app/src/pages/PosAdmin/ProductManagement/index.tsx`
  - 为“商品类型”列增加 `sorter: true`
  - 增加 `filterDropdown`（三个选项：0 普通商品、1 套装商品、2 多条码商品），选择后触发表格刷新，通过现有 `columnFilters` 机制带到后端（为统一建议：在生成 `params` 时将 `columnFilters['productType']` 的值放入 `params.productType`）。

## 前端改动（条码列点击显示 Code128 图片）
- 依赖：在 `ReactUmi/my-app` 增加 `jsbarcode` 依赖，用于前端生成 Code128 条码。
- 交互：在条码列 `render` 内包裹可点击元素（如 `<Button type="link">{barcode}</Button>`）。
- 弹窗：新增 `Modal` 与本地状态 `barcodeModalVisible`、`currentBarcode`；在弹窗中渲染一个 `<svg ref={svgRef}>`。
- 渲染：`import JsBarcode from 'jsbarcode'`；在弹窗 `open` 时用 `JsBarcode(svgRef.current, currentBarcode, { format: 'CODE128', displayValue: true })` 生成条码。
- 边界：空条码不显示；对超长或含非 ASCII 的值进行校验并提示。

## 备选方案（后端生成图片）
- 如不希望引入前端库，可在后端增加生成 Code128 的接口（例如集成 `BWIP-JS` 或 .NET 条码库），前端直接 `<img src={url}>` 显示；但开发成本更高，此次优先前端生成。

## 验证要点
- 商品类型筛选：选择后列表正确过滤；清空后恢复。
- 商品类型排序：点击列头升降序正确；与后端排序一致。
- 条码弹窗：点击条码文本打开弹窗，显示 Code128 图像及条码值；关闭后销毁或复用渲染。
- 兼容性：无条码或非法值时提示或显示 `-`；不影响既有搜索与分页。

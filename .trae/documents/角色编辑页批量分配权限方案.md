# 后端动态导航权限方案（修订版）

## 现状分析

当前导航菜单完全由前端 [routes.tsx](file:///Users/sean/DEV/hbweb_rv/src/router/routes.tsx) 硬编码，每个菜单项通过 `accessKey` 指向 `AccessControl` 上的布尔属性。前端 [access.ts](file:///Users/sean/DEV/hbweb_rv/src/utils/access.ts) 根据 `currentUser.permissions[]` + `roleNames[]` 计算这些布尔值。

**三大问题**：

1. 添加新页面/修改权限需要在前后端同时改代码
2. 权限粒度过粗——`canManageStore` 同时控制 6 个页面，仓库商品和分店商品混用同一个 `Warehouse.Manage`
3. 定价策略/促销管理硬编码为 `isAdmin`，无法下放给非 Admin 角色

***

## 实施步骤

### 步骤 1：更新 Permissions.cs 新增权限码

**文件**：`BlazorApp.Shared/Constants/Permissions.cs`

* 新增 `StoreProducts`, `Promotions`, `PricingStrategy`, `LocalPurchase`, `AustralianSuppliers`, `Store` 静态类

* 更新 `GetAllPermissions()` 种子数据

### 步骤 2：更新前端 AccessControl 类型和计算

**文件**：`hbweb_rv/src/types/auth.ts` — 新增 accessKey 布尔属性
**文件**：`hbweb_rv/src/utils/access.ts` — 新增计算逻辑

### 步骤 3：更新前端路由权限绑定

**文件**：`hbweb_rv/src/router/routes.tsx` — 替换旧的 accessKey 为新的细粒度 accessKey

### 步骤 4：创建后端导航服务

**文件**：`BlazorApp.Api/Services/NavigationService.cs`

* 硬编码完整导航树（与前端 routes.tsx 结构一致）

* `Permission` 字段使用新权限码

* 递归过滤：Admin 全开，其他按 `permissions[]` 匹配

### 步骤 5：创建后端导航控制器

**文件**：`BlazorApp.Api/Controllers/NavigationController.cs`

* `GET /api/navigation/menu` — 返回当前用户可见菜单树

### 步骤 6：前端接入后端菜单

**文件**：`hbweb_rv/src/services/navigationService.ts`（新建）
**文件**：`hbweb_rv/src/store/auth.ts` — login/fetchCurrentUser 后拉取菜单
**文件**：`hbweb_rv/src/router/routes.tsx` — `buildMenus()` 优先用后端返回的菜单

### 步骤 7：编译 + Docker 部署

1. `dotnet build` 后端
2. `npx tsc && npm run build` 前端
3. `docker compose up -d --build` 两边部署

***

## 风险

| 风险                     | 应对                                           |
| ---------------------- | -------------------------------------------- |
| 路由 guard 仍用旧 accessKey | 步骤 3 同步更新 routes.tsx 中所有 accessKey           |
| 后端菜单与前端路由不同步           | 后端 NavigationService 复制 routes.tsx 完整结构，逐项校对 |
| 已有角色权限缺失               | 部署后需在角色管理页为各角色分配新权限码                         |

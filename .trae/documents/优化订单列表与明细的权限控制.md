**实施计划：权限控制优化**

1. **用户角色与权限定义**:
   * 前端已通过 `src/access.ts` 定义了 `isAdmin`、`isManager` 等权限。
   * 我们将使用 `useAccess` hook 在组件中获取这些权限状态。
   * 假设“普通用户”对应于非管理员（`!access.isAdmin`）且可能具有 `isStoreStaff` 或 `isUser` 角色。我们将主要使用 `!access.isAdmin` 来界定“普通用户”的限制。

2. **Task 1: 订单列表 (OrderList)**
   * **目标**: 普通用户不显示“删除”按钮。
   * **文件**: `src/pages/StoreOrder/OrderList/index.tsx`
   * **修改**:
     * 引入 `useAccess` hook。
     * 在 `columns` 定义中，修改 `Actions` 列的 `render` 函数。
     * 只有当 `access.isAdmin` 为 `true` 时，才渲染 `<Popconfirm>` 删除按钮。

3. **Task 2: 订单明细 (OrderDetails)**
   * **目标**: 普通用户只能编辑“购物车”状态（FlowStatus=0）的订单，其他状态不能编辑。
   * **文件**: `src/pages/StoreOrder/OrderDetails/index.tsx`
   * **修改**:
     * 引入 `useAccess` hook。
     * 引入 `useModel` 获取 `initialState` (可选，`useAccess` 已足够)。
     * 定义一个计算属性 `isEditable`:
       * 如果 `access.isAdmin` 为 `true`，则始终为 `true` (或根据业务需求，管理员可能也有状态限制，但用户需求仅提到普通用户受限。通常管理员可以编辑任何状态，或者我们可以保守一点，只限制普通用户)。
       * **用户需求细化**: "普通用户 只能 编辑 购物状态的订单"。这意味着：
         * `isEditable = access.isAdmin || (order.flowStatus === 0)`
     * **应用 `isEditable` 控制**:
       * **备注修改**: 禁用 Input 和 Save 按钮。
       * **快速添加 (Quick Add)**: 隐藏整个 Form。
       * **批量选择 (Batch Select)**: 隐藏按钮。
       * **数量修改 (InputNumber)**: 设置 `disabled={!isEditable}`。
       * **删除行 (Actions)**: 隐藏删除按钮。

4. **验证**:
   * 检查代码逻辑是否正确覆盖了所有编辑入口。
   * 确保管理员权限下功能不受影响。

5. **执行顺序**:
   * 修改 `OrderList/index.tsx`。
   * 修改 `OrderDetails/index.tsx`。

*(注：由于后端 API 层面可能也需要做相应的权限校验以确保安全，但用户当前请求明确针对前端 UI 显示。如果需要后端校验，可以在后续步骤中添加)*
